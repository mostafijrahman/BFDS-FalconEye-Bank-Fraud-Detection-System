PROJECT PROPOSAL
Bank Fraud Detection System (BFDS)

Document Version: 1.0  
Date: 9th April 2026  

EXECUTIVE SUMMARY

The Bank Fraud Detection System (BFDS) is a comprehensive database-driven solution designed to identify, flag, 
and investigate suspicious financial transactions in real-time. This system combines advanced SQL analytics, 
database automation, and security principles to prevent fraud losses while minimizing false positives 
that impact customer experience. 
 
The three main rules for fraud detection are: 
 
### Rule 1: Geographic Impossibility (Impossible Travel)

**Scenario:** A transaction occurs at location A, then another at location B that would require traveling faster 
than commercial airline speeds.

### Rule 2: Structured Transaction Patterns

**Scenario:** Multiple small transactions occurring at unusual times or patterns that suggest coordinated fraud. Anti-money laundering (AML) detection.

### Rule 3: Blacklisted Merchant / High-Risk Category

**Scenario:** A transaction occurs at a known high-risk merchant or in a high-risk category (online gambling, wire transfers).

**Core Entities:**
1. **Users** - Bank account holders
2. **Cards** - Credit/debit cards linked to users
3. **Merchants** - Businesses accepting card payments
4. **Transactions** - Individual payment records
5. **Alerts** - Fraud flags generated by rules
6. **User Devices** - Devices linked to user accounts
7. **User Locations** - Historical geographic locations
8. **Authentication** - User authentication(user_id) and authorization(password) for the Bank Fraud Detection System (BFDS) users - Bank EMployees


**Relationships:**
- User has many Cards (1:N)
- Card has many Transactions (1:N)
- User has many Transactions (1:N)
- Merchant has many Transactions (1:N)
- Transaction may generate Alert (1:0..1)


Row-Level Security (RLS):
1] Super Admin - Can view all data
2] Compliance Officer- Can read reports, Transaction logs and Alert tables
3] Bank Employee - Can view user data, cards, transactions, alerts and user devices

*Need testing query for all three rules in the from of transaction data
